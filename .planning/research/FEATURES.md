# Feature Research

**Domain:** personal website + resume surface
**Researched:** 2026-03-24
**Confidence:** HIGH

## Feature Landscape

### Table Stakes (Users Expect These)

Features users assume exist. Missing these = the portfolio feels incomplete.

| Feature | Why Expected | Complexity | Notes |
|---------|--------------|------------|-------|
| Clear hero with name, title, location, and current availability | Visitors need instant context for who this is and what role is wanted | LOW | Already present in `frontend/pages/Landing.jsx` with name, role, “Open to roles and contracts,” headshot, and location. |
| Experience history with dates, titles, employers, and bullets | A resume-style site must prove real work history quickly | MEDIUM | Already present in `frontend/pages/Landing.jsx` and mirrored in `resume_pdf_generator.py` under `POSITIONS`. |
| Project highlights with links and outcome-oriented summaries | Portfolio visitors expect proof of shipped work | MEDIUM | Already present in `frontend/pages/Landing.jsx` and mirrored in `resume_pdf_generator.py` under `PROJECTS`. |
| Skills / stack summary | Quick scanability matters more than exhaustive detail | LOW | Already present in `frontend/pages/Landing.jsx` and `resume_pdf_generator.py` (`SKILLS`). |
| Downloadable resume PDF | Public portfolio/resume pages are expected to have an easy offline/shareable version | MEDIUM | Already present via `frontend/utils/Resume.jsx`, which downloads `Torrin_Leonard_Resume.pdf` from `frontend/public/`; generated by `resume_pdf_generator.py`. |
| Contact path (form + direct contact links) | A portfolio without a clear next step feels unfinished | MEDIUM | Already present in `frontend/utils/Contact.jsx`, plus the landing page buttons for email and GitHub in `frontend/pages/Landing.jsx`. |
| Content parity between landing page and PDF | This project’s core promise is a consistent public story across formats | MEDIUM | Explicit project requirement in `.planning/PROJECT.md`; source overlap is visible between `frontend/pages/Landing.jsx` and `resume_pdf_generator.py`. |

### Differentiators (Competitive Advantage)

Features that set the product apart. Not required, but valuable.

| Feature | Value Proposition | Complexity | Notes |
|---------|-------------------|------------|-------|
| Single-story branding across web + PDF | Makes the site feel deliberate instead of like two separate documents | MEDIUM | The same experience/projects/skills narrative is maintained in `frontend/pages/Landing.jsx` and `resume_pdf_generator.py`. |
| Strong technical positioning (AI, vector search, accessibility) | Signals the exact kind of work this engineer wants | LOW | The hero and summaries emphasize AI-powered web apps, vector search pipelines, and WCAG in both sources. |
| Polished card layout with logos, date ranges, and external links | Improves scanability and makes the portfolio feel curated | MEDIUM | Implemented in the reusable `Card` component in `frontend/pages/Landing.jsx`. |
| Multi-channel contact affordances | Lets visitors choose form, email, or GitHub without friction | LOW | Landing page has `Contact`, `Email`, and GitHub buttons; contact form is in `frontend/utils/Contact.jsx`. |
| “Resume” button with immediate download feedback | Reduces ambiguity and feels responsive | LOW | `frontend/utils/Resume.jsx` swaps to “Downloaded!” with a check icon after click. |

### Anti-Features (Commonly Requested, Often Problematic)

Features that seem good but create problems.

| Feature | Why Requested | Why Problematic | Alternative |
|---------|---------------|-----------------|-------------|
| Separate CMS-managed copy for landing page and resume | Feels easier to edit in one place | Creates drift and undermines the site’s main promise | Keep `frontend/pages/Landing.jsx` and `resume_pdf_generator.py` as paired sources of truth. |
| Blog/platform-style expansion | Makes the site feel richer | Pulls focus away from the portfolio/resume goal | Keep the landing page concise and professional. |
| Heavy animations or gimmicky interactions | Looks “modern” | Can distract from hiring signals and hurt clarity | Use subtle polish only (the current shine/feedback pattern is enough). |
| Overlong project writeups | Seems like more detail = better | Buries the core story and hurts scannability | Keep bullets tight and outcome-oriented. |
| Inconsistent wording between web and PDF | Appears harmless during edits | Breaks trust when recruiters compare both artifacts | Update both outputs together every time wording changes. |
| Extra product-like features (accounts, dashboards, complex settings) | Tempting for experimentation | Overbuilds a personal site into something it is not | Stay focused on presentation, contact, and proof of work. |

## Feature Dependencies

```text
[Consistent public story]
    └──requires──> [Shared content structure]
                       └──requires──> [Landing page + PDF generator parity]

[Resume download]
    └──requires──> [Generated PDF in `frontend/public/`]

[Contact CTA]
    └──requires──> [Working `/contact` submission path in `frontend/utils/Contact.jsx`]

[Strong portfolio scanability]
    ──enhances──> [Experience / projects / skills sections]
```

### Dependency Notes

- **Consistent public story requires shared content structure:** the same experience, project, and skills facts need to live in both `frontend/pages/Landing.jsx` and `resume_pdf_generator.py`.
- **Resume download requires generated PDF in `frontend/public/`:** `frontend/utils/Resume.jsx` points at `Torrin_Leonard_Resume.pdf`, so the build flow must keep that asset current.
- **Contact CTA requires a working `/contact` path:** the form in `frontend/utils/Contact.jsx` posts to `/contact`, so the public site depends on a real submission endpoint.

## MVP Definition

### Launch With (v1)

Minimum viable product — what's needed for a clean public portfolio/resume experience.

- [ ] Hero with name, role, location, and availability — immediate identity and intent (`frontend/pages/Landing.jsx`)
- [ ] Experience / projects / skills sections — the proof of work and scannability (`frontend/pages/Landing.jsx`, `resume_pdf_generator.py`)
- [ ] Resume PDF download — shareable, recruiter-friendly artifact (`frontend/utils/Resume.jsx`, `resume_pdf_generator.py`)
- [ ] Contact options — form plus direct links (`frontend/utils/Contact.jsx`, `frontend/pages/Landing.jsx`)
- [ ] Web/PDF content sync — avoid public drift (`.planning/PROJECT.md`)

### Add After Validation (v1.x)

Features to add once core is working.

- [ ] Better update workflow docs — reduce the risk of landing/PDF divergence (`.planning/PROJECT.md`)
- [ ] Subtle polish improvements — only if they do not slow the scan path (`frontend/pages/Landing.jsx`)

### Future Consideration (v2+)

Features to defer until the core portfolio story is stable.

- [ ] Rich CMS/editor workflow — only if manual sync becomes painful
- [ ] Additional public content sections — only if they support hiring signal

## Competitor Feature Analysis

| Feature | Competitor A | Competitor B | Our Approach |
|---------|--------------|--------------|--------------|
| Resume availability | Usually a static PDF link | Often absent or buried | Make it prominent with `frontend/utils/Resume.jsx`. |
| Portfolio proof | Case studies or project cards | Often sparse | Use concise work + project cards in `frontend/pages/Landing.jsx`. |
| Contact path | Contact form or social links | Sometimes email only | Offer form, email, and GitHub together. |
| Content consistency | Web and PDF often diverge | Same | Keep the same story in `frontend/pages/Landing.jsx` and `resume_pdf_generator.py`. |
